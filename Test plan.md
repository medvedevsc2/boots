<a name="introduction"/>

## Введение

Данный документ описывает план тестирования приложения "OnlineSneakersStore". Документ предназначен для людей, выполняющих тестирование данного проекта. Цель тестирования - проверка соответствия реального поведения программы проекта и ее ожидаемого поведения, которое описано в требованиях.

<a name="items"/>

## Объект тестирования

В качестве объектов тестирования можно выделить следующие функциональные требования:

* Регистрация и авторизация пользователей;
* Просмотр каталога кроссовок;
* Поиск и фильтрация товаров;
* Управление корзиной покупок;
* Оформление заказов;
* Управление товарами (для менеджеров);
* Управление заказами (для менеджеров);
* Управление пользователями (для администраторов).

<a name="quality"/>

## Атрибуты качества

1. Функциональность:
    - функциональная полнота: приложение должно выполнять все заявленные функции;
    - функциональная корректность: приложение должно выполнять все заявленные функции корректно;
2. Удобство использования:
    - все функциональные элементы пользовательского интерфейса имеют названия, описывающие действие, которое произойдет при выборе элемента;
    - минимальное количество шагов для выполнения основных операций.
3. Производительность:
    - время отклика системы не должно превышать 3 секунд;
    - система должна выдерживать одновременную работу multiple пользователей.

<a name="risk"/>

## Риски

В данном случае к рискам можно отнести:
* проблемы с интеграцией платежной системы;
* ошибки при обработке больших объемов данных о товарах;
* проблемы с отображением на разных устройствах и браузерах;
* возможные утечки данных пользователей.

<a name="features"/>

## Аспекты тестирования

В ходе тестирования планируется проверить реализацию основных функций приложения. Аспекты, подвергаемые тестированию: 
* Регистрация и авторизация пользователей  
* Просмотр каталога товаров  
* Поиск и фильтрация товаров  
* Управление корзиной покупок  
* Оформление заказа  
* Управление товарами  
* Управление заказами  
* Управление пользователями  

### Регистрация и авторизация пользователей
Этот вариант использования необходимо протестировать на:
* успешную регистрацию нового пользователя;
* обработку некорректных данных при регистрации;
* успешную авторизацию существующего пользователя;
* обработку неверных учетных данных;
* восстановление пароля.

### Просмотр каталога товаров
Этот вариант использования необходимо протестировать на:
* отображение всего каталога товаров;
* отображение детальной информации о товаре;
* корректное отображение цен и наличия;
* пагинацию при большом количестве товаров.

### Поиск и фильтрация товаров
Этот вариант использования необходимо протестировать на:
* поиск товаров по названию и бренду;
* фильтрацию по размеру, цене, цвету;
* сброс фильтров;
* обработку пустых результатов поиска.

### Управление корзиной покупок
Этот вариант использования необходимо протестировать на:
* добавление товаров в корзину;
* изменение количества товаров в корзине;
* удаление товаров из корзины;
* подсчет общей стоимости;
* сохранение корзины между сессиями.

### Оформление заказа
Этот вариант использования необходимо протестировать на:
* заполнение данных доставки;
* выбор способа оплаты;
* подтверждение заказа;
* обработку ошибок оплаты;
* отправку уведомлений о заказе.

### Управление товарами
Этот вариант использования необходимо протестировать на:
* добавление новых товаров;
* редактирование информации о товарах;
* обновление остатков на складе;
* удаление товаров;
* загрузку изображений товаров.

### Управление заказами
Этот вариант использования необходимо протестировать на:
* просмотр списка всех заказов;
* изменение статусов заказов;
* фильтрацию заказов по статусу и дате;
* просмотр детальной информации о заказе.

### Управление пользователями
Этот вариант использования необходимо протестировать на:
* просмотр списка пользователей;
* блокировку/разблокировку пользователей;
* смену ролей пользователей;
* просмотр истории заказов пользователя.

## Нефункциональные требования:
* все функциональные элементы пользовательского интерфейса имеют названия, описывающие действие, которое произойдет при выборе элемента;
* время отклика системы не превышает 3 секунд;
* корректное отображение на основных браузерах и мобильных устройствах.

<a name="approach"/>

## Подходы к тестированию

При тестировании будет использован ручной подход. Тестирование будет проводиться на разных уровнях:
* Модульное тестирование - тестирование отдельных компонентов;
* Интеграционное тестирование - тестирование взаимодействия между компонентами;
* Системное тестирование - тестирование системы в целом;
* Приемочное тестирование - тестирование на соответствие требованиям заказчика.

<a name="pass"/>

## Представление результатов

Результаты тестирования будут представлены в документе "Результаты тестирования", который будет включать:
* описание тестовых сценариев;
* ожидаемые результаты;
* фактические результаты;
* статус теста (пройден/не пройден);
* обнаруженные дефекты с приоритетами.

<a name="conclusion"/>

## Выводы

Данный тестовый план позволяет протестировать основной функционал интернет-магазина кроссовок. Успешное прохождение всех тестов не гарантирует полной работоспособности на всех версиях платформ и архитектурах, однако позволяет полагать, что данное программное обеспечение работает корректно. Тестирование должно проводиться на регулярной основе в процессе разработки и перед выпуском каждой новой версии приложения.
